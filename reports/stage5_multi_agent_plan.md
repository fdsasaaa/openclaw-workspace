## 三、容器化评估

### 3.1 容器化利弊

| 方面 | 优势 | 劣势 |
|------|------|------|
| **隔离性** | 实例间不互相影响 | 增加资源开销 |
| **可移植** | 跨平台部署 | Windows Docker体验一般 |
| **版本控制** | 环境一致性 | 配置复杂 |
| **资源管理** | 可限制CPU/内存 | 需要额外管理 |

### 3.2 当前建议

**不建议当前阶段容器化**

原因:
- OpenClaw设计为本地守护进程模式
- Windows下Docker Desktop资源占用高
- 单用户场景下容器化收益有限

**替代方案**: 进程级隔离
- 使用不同agent目录
- 通过 `openclaw agents add` 创建
- 共享同一Gateway

## 四、最小协作流程

### 4.1 标准协作流程

```
1. 任务接收
   ↓
2. 协调器拆解 → 生成子任务清单
   ↓
3. 分发执行 → 子实例并行处理
   ↓
4. 结果收集 → 等待全部完成
   ↓
5. 汇总整合 → 协调器合并输出
   ↓
6. 质量检查 → 验证一致性
   ↓
7. 交付用户
```

### 4.2 子任务分发格式

```json
{
  "taskId": "task-20260305-001",
  "parentTask": "撰写技术文档",
  "subtasks": [
    {
      "subtaskId": "sub-001",
      "agent": "coding-agent",
      "prompt": "生成代码示例：Python文件读取",
      "expectedOutput": "code-block",
      "timeout": 300
    },
    {
      "subtaskId": "sub-002",
      "agent": "researcher",
      "prompt": "检索Python文件处理最佳实践",
      "expectedOutput": "bullet-points",
      "timeout": 300
    },
    {
      "subtaskId": "sub-003",
      "agent": "writer",
      "prompt": "整合代码和最佳实践，撰写说明文档",
      "inputs": ["sub-001", "sub-002"],
      "expectedOutput": "markdown",
      "timeout": 600
    }
  ]
}
```

## 五、模板文件结构

### 5.1 推荐模板清单

当创建新子实例时，需准备以下模板：

| 模板文件 | 用途 | 关键内容 |
|----------|------|----------|
| `identity.md` | 身份定义 | 名字、角色、emoji、专长 |
| `soul.md` | 行为准则 | 工作风格、边界、输出要求 |
| `user.md` | 用户信息 | 用户偏好、沟通方式 |
| `tools.md` | 工具清单 | 可用工具、调用方式 |
| `memory.md` | 记忆初始化 | 相关背景知识、历史上下文 |

### 5.2 identity.md 模板

```markdown
# IDENTITY.md

- **Name**: [子实例名称，如 coding-agent]
- **Creature**: AI 助手 / [协调器名称]的子实例
- **Vibe**: [专业、高效、严谨]
- **Emoji**: [🧑‍💻/🔍/✍️]
- **Specialty**: [代码/研究/写作]
- **Parent**: [协调器名称，如 虾哥]

## Scope
- 负责: [具体职责范围]
- 不负责: [明确边界]
- 必须汇报: [何时需上报协调器]

## Output Format
- 代码: [语言、风格、注释要求]
- 文档: [格式、长度、结构]
- 汇报: [简明摘要格式]
```

### 5.3 soul.md 模板

```markdown
# SOUL.md

## Core Principles
- [专注单一任务，不发散]
- [输出可直接使用，不半成品]
- [遇到问题立即上报，不猜测]

## Boundaries
- 不直接修改用户文件
- 不执行未经确认的危险操作
- 不保留长期记忆（检查点除外）

## Communication
- 向协调器汇报时使用: [格式]
- 任务完成时输出: [结果摘要格式]
```

### 5.4 user.md 模板


## 七、冲突规避规则

### 7.1 记忆污染防范

| 风险 | 规则 | 检查方式 |
|------|------|----------|
| 子实例A记忆混入子实例B | 每个子实例独立workspace目录 | 启动时检查路径 |
| 长期记忆冲突 | 子实例不写MEMORY.md，只写检查点 | 权限控制 |
| 上下文溢出 | 单任务完成后强制清理 | token计数 |

### 7.2 端口冲突防范

| 服务 | 默认端口 | 分配规则 |
|------|----------|----------|
| Gateway | 18789 | 全局唯一，所有实例共享 |
| Canvas Host | 18793 | 全局唯一 |
| 子实例服务 | 动态 | 协调器统一分配 |

### 7.3 文件冲突防范

| 场景 | 规则 |
|------|------|
| 同时写入同一文件 | 加锁机制或协调器序列化 |
| 临时文件命名 | 使用`temp-{taskId}-{timestamp}` |
| 输出目录 | 子实例按ID分区 |

### 7.4 权限失控防范

| 风险 | 规则 |
|------|------|
| 子实例越权操作 | 子实例只读核心配置，不修改 |
| 敏感信息泄露 | 子实例不接触auth-profiles.json |
| 删除操作 | 子实例无删除权限，需协调器确认 |
